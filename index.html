⚡ Як працює реактивність

Fine-grained reactivity — серце Solid.js

Дві моделі реактивності

React — Pull

  1. setState()
  2. React перезапускає render()
  3. Будує новий Virtual DOM
  4. Порівнює з попереднім (diffing)
  5. Патчить реальний DOM

Компонент — функція, яка викликається знову і знову

Solid — Push

  1. setSignal()
  2. Solid знаходить підписників
  3. Оновлює тільки їх
  4. DOM патчиться напряму

Компонент — фабрика, що запускається один раз

💡 Уявіть Excel: формула =A1*2 у клітинці B1 — B1 «підписана» на A1. Змінили A1 → B1 перерахувалась автоматично. Solid робить те саме для DOM.

Реактивний граф

Кожна стрілка — підписка. Оновлення йде тільки по підписках.

Змінився count → оновились тільки підписані вузли. Решта DOM не чіпається.

Як Solid відстежує залежності

Спрощена реалізація під капотом — жодної магії

// Глобальна змінна — «поточний спостерігач»
let currentObserver = null;

function createSignal(value) {
  const subscribers = new Set();

  const read = () => {
    if (currentObserver) {
      subscribers.add(currentObserver); // ← реєструємо підписника
    }
    return value;
  };

  const write = (newValue) => {
    value = newValue;
    subscribers.forEach(sub => sub()); // ← повідомляємо всіх
  };

  return [read, write];
}

function createEffect(fn) {
  const effect = () => {
    currentObserver = effect; // ← «слухаємо»
    fn();                     // ← усі count() всередині підпишуться
    currentObserver = null;   // ← прибираємо слухача
  };
  effect(); // перший запуск
}
💡 count() — це не просто читання значення. Це реєстрація залежності у поточному контексті.

createSignal — деталі

const [count, setCount] = createSignal(0);

// Читання — завжди функція
count()   // ✅ 0
count     // ❌ [Function] — не значення!

// Setter — два варіанти
setCount(5);                    // нове значення напряму
setCount(prev => prev + 1);    // updater-функція

// Нюанс: рівність
setCount(5);
setCount(5); // ефект НЕ спрацює — 5 === 5, нічого не змінилось

// Для об'єктів — посилання різні → завжди тригер
const [obj, setObj] = createSignal({ x: 1 });
setObj({ x: 1 }); // тригерить! {} !== {}

// Вимкнути порівняння:
createSignal(0, { equals: false }); // завжди тригерить

createMemo — обчислюваний стан

const [firstName, setFirstName] = createSignal('Dmytro');
const [lastName, setLastName]   = createSignal('Kovalenko');

// ✅ createMemo — кешується, без масиву залежностей
const fullName = createMemo(() => `${firstName()} ${lastName()}`);
fullName() // 'Dmytro Kovalenko'

// Перераховується тільки при зміні firstName або lastName
// Якщо залежності не змінились — повертає кеш

createMemo потрібен коли:

  • Обчислення дороге
  • Результат читається у кількох місцях
  • Хочемо уникнути повторних розрахунків

Проста функція підійде якщо:

// Теж реактивно в JSX!
const fullName = () =>
  `${firstName()} ${lastName()}`;

// <p>{fullName()}</p> — ✅ працює

createEffect — нюанси

Cleanup

createEffect(() => {
  const timer = setInterval(
    () => console.log(count()),
    1000
  );
  // Викликається перед кожним
  // перезапуском і при знищенні
  onCleanup(() => clearInterval(timer));
});

Нескінченний цикл

// ❌ НЕ РОБІТЬ ТАК
createEffect(() => {
  setCount(count() + 1); // 💥 loop!
});

// ✅ Для похідного стану
const nextCount = createMemo(
  () => count() + 1
);

async пастка — після await відстеження зупиняється!

// ❌ userId після await — НЕ відстежується
createEffect(async () => {
  const data = await fetch(`/api/${userId()}`); // ← після await!
});

// ✅ Читаємо сигнал ДО await
createEffect(() => {
  const id = userId(); // ← тут відстежується
  void (async () => {
    const data = await fetch(`/api/${id}`);
    setUser(await data.json());
  })();
});

Де сигнали відстежуються?

✅ Реактивний контекст

  • createEffect(() => { ... })
  • createMemo(() => { ... })
  • JSX: <p>{"{count()}"}</p>
  • Тіло компонента (перший рендер)

❌ НЕ реактивний контекст

  • setTimeout / setInterval
  • addEventListener callbacks
  • Після першого await
  • Звичайні функції поза контекстом
// ❌ Читання без підписки
function logCount() {
  console.log(count()); // виведе один раз і забуде
}
setTimeout(logCount, 1000);

// ✅ Читаємо в ефекті, ДО setTimeout
createEffect(() => {
  const current = count(); // підписка є
  setTimeout(() => console.log(current), 1000);
});

⚠ Пастка №1 — деструктуризація props

// ❌ НІКОЛИ ТАК — реактивність зникає!
const MyComponent = ({ name, age }) => {
  // name і age — звичайні значення, знімок при першому рендері
  return <p>{name} is {age}</p>;
};
// Змінили name зовні — компонент не оновиться

Варіант 1 — читати через props

const MyComponent = (props) => {
  return (
    <p>{props.name} is {props.age}</p>
  );
};

Варіант 2 — splitProps

const MyComponent = (props) => {
  const [local, rest] = splitProps(
    props, ['name', 'age']
  );
  // local.name — геттер, не примітив
  return <p>{local.name} is {local.age}</p>;
};
💡 props у Solid — це Proxy з геттерами. Деструктуризація читає значення один раз і зберігає примітив. Реактивність губиться.
Для дефолтних значень: mergeProps({ age: 18 }, props)

createStore — складний стан

const [state, setState] = createStore({
  user: { name: 'Dmytro', address: { city: 'Kyiv' } },
  items: [1, 2, 3],
});

// Читання — як звичайний об'єкт (але реактивно)
state.user.name          // 'Dmytro'
state.user.address.city  // 'Kyiv'

// Запис через шлях
setState('user', 'name', 'Ivan');
setState('user', 'address', 'city', 'Lviv');

// Масиви
setState('items', items => [...items, 4]);
setState('items', 1, 99); // items[1] = 99
Гранулярність: кожне поле — окремий реактивний вузол. Компонент, що читає state.user.name, не оновиться при зміні state.user.address.city.
Та сама пастка: 
// ❌
const { name } = state.user; // рядок!

// ✅
state.user.name // геттер, реактивно

batch та untrack

batch — групування оновлень

const [a, setA] = createSignal(1);
const [b, setB] = createSignal(2);

createEffect(() => console.log(a() + b()));

// Без batch — ефект спрацює ДВІЧІ
setA(10);
setB(20);

// З batch — ефект спрацює ОДИН РАЗ
batch(() => {
  setA(10);
  setB(20);
});

У обробниках DOM-подій Solid батчить автоматично. batch потрібен в async-коді.

untrack — читання без підписки

const [a, setA] = createSignal(1);
const [b, setB] = createSignal(2);

createEffect(() => {
  // Ефект реагує на a()
  // але НЕ реагує на b()
  console.log(a() + untrack(b));
});
Корисно коли b — допоміжне значення, яке не повинно тригерити перерахунок.

Lifecycle: onMount / onCleanup

const ChartComponent = () => {
  let canvasRef: HTMLCanvasElement | undefined;
  let chartInstance: Chart | undefined;

  onMount(() => {
    // Виконується після вставки у DOM (аналог useEffect(fn, []))
    // ⚠ НЕ запускається на сервері (SSR)
    chartInstance = new Chart(canvasRef!, { ... });
  });

  onCleanup(() => {
    // Виконується при знищенні компонента
    // Аналог return () => { ... } у useEffect
    chartInstance?.destroy();
  });

  return <canvas ref={canvasRef} />;
};
onCleanup можна також викликати всередині createEffect — там він прибирає підписки перед кожним перезапуском ефекту.

🧱 Ключові примітиви

Show · For · Index · createResource

Show, For, Index — чому не if/.map()

Show — умовний рендер

// ✅ Show — оптимізований
<Show when={isLoggedIn()} fallback={<Login />}>
  <Dashboard />
</Show>

// ⚠ Тернарний оператор — теж працює,
// але Solid не оптимізує так само
{isLoggedIn() ? <Dashboard /> : <Login />}

For — реактивний список

// ✅ For — key-diffing, не перебудовує все
<For each={items()}>
  {(item, index) => (
    <li>{index()} — {item.name}</li>
    // index — теж сигнал-функція!
  )}
</For>

// ❌ .map() — не реактивний,
// перебудовує весь список
{items().map(i => <li>{i.name}</li>)}
Index (альтернатива For) — стабілізує за індексом, а не за елементом. Підходить для примітивних масивів (числа, рядки). 
<Index each={numbers()}>
  {(number, i) => <span>{number()}</span>}
  {/* number — сигнал, i — звичайне число */}
</Index>

createResource — async дані

// Базовий варіант
const [posts] = createResource(fetchPosts);

// З динамічним параметром
// Перезапускається при зміні userId()
const [user] = createResource(userId, (id) => fetchUser(id));

// Якщо source повертає false/undefined — fetch не викликається
const [post] = createResource(
  () => props.id || false,
  (id) => fetchPost(id)
);

// Стани
user()          // значення або undefined
user.loading    // boolean
user.error      // помилка або undefined
user.state      // 'unresolved'|'pending'|'ready'|'refreshing'|'errored'

// Примусове оновлення
const [data, { refetch, mutate }] = createResource(fetchPosts);
refetch();           // повторний fetch
mutate(newValue);    // оновити локально без fetch
// Разом із Suspense
<Suspense fallback={<Spinner />}>
  <p>{user()?.name}</p>
</Suspense>

Server Functions

// query — читання (GET-семантика, кешується)
const getUser = query(async (id: string) => {
  'use server';
  return await db.users.findById(id);
}, 'getUser');

// action — мутації (POST-семантика)
const createPost = action(async (formData: FormData) => {
  'use server';
  const title = formData.get('title') as string;
  return await db.posts.create({ title });
});

// Використання у компоненті
const MyPage = () => {
  const user = createAsync(() => getUser(params.id));

  return (
    <form action={createPost} method="post">
      <input name="title" />
      <button type="submit">Створити</button>
    </form>
  );
};
Server functions виконуються на сервері, але викликаються як звичайні функції у клієнтському коді. Solid автоматично генерує RPC-шар.

Зведення підводних каменів

⚠️
count — функція, не значення. Ніколи не деструктуруйте props!
⚠️
Компонент рендериться один раз. Логіка в тілі виконується тільки при ініціалізації.
⚠️
Читання сигналу поза реактивним контекстом — підписки немає.
⚠️
Після await у createEffect — відстеження залежностей зупиняється.
⚠️
setSignal всередині createEffect на той самий сигнал → нескінченний цикл.
⚠️
Масиви у createStore — не мутуйте напряму, використовуйте setState.
⚠️
<For>, а не .map() — інакше втрачаєте key-diffing.
⚠️
JSX компілюється інакше — не вставляйте Solid-компоненти у React-проєкт.

Коли обирати Solid?

✅ Підходить

  • Новий проєкт без legacy-коду
  • Важлива продуктивність
  • Real-time / дашборди
  • Мобільний веб з обмеженим трафіком
  • Команда готова вивчити новий API

❌ Не підходить

  • Критична велика екосистема React-бібліотек
  • Команда тільки починає з фронтенду
  • Великий legacy React-проєкт
